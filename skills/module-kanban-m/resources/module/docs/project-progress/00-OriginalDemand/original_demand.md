# 原始需求

## 背景
我们已经拥有一个app 可以访谈用户, 并生成一些关于他们生活的记忆卡片, 之后我们会使用这些记忆卡片构建他们的数字分身
我们现在的任务是获取访谈过程中拿到的成品记忆卡片, 并为数字分身Agent 提供合适的记忆卡片检索mcp

## 核心需求

我们的核心诉求是编写一个记忆卡片管理mcp 并交付给对外的大模型Agent 调用 

## 功能点

我们计划做两个服务。FastAPI 支撑的HTTP 服务 
支持 记忆卡片增加, 记忆卡片删除, 记忆卡片更新 等操作
相关信息 可以接入 apifox mcp 去获取

另一个服务是 fastmcp 提供的mcp-http 服务
只支持 记忆卡片的查询功能, 输入用户提问(字符串) 输出 检出的记忆卡片列表(字符串)

## 技术栈

FastAPI + Fastmcp + qdrant + llama-index

## 基本的向量存储结构

metadata : 
    {} // 标识不同数字分身对于这个记忆卡片的可见性
    // 支持是否删除
    // user-id

通过filter 技术来过滤掉不可见的数据, 只在可见数据中进行向量化检索




## 注意点
一个用户只有一个访谈, 维护大量(100-500)个记忆卡片, 但一个用户可以拥有多个数字分身(代号小智1, 小智2) 数字分身的上限是100个
所有接口都要求有鉴权
使用llama-index 中的高级检索优化技术来不断优化召回率和准确度
